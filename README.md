# 项目开发规范与行为准则

##  AI 助手行为准则

### 提示词处理
- 对输入的提示词进行适当优化后再处理，确保理解准确
- 如对提示词有异议或存在歧义，必须明确指出，不得盲目生成
- 保持沟通透明，遇到不确定的地方主动询问确认

### 开发原则
- 禁止编写 TODO 注释，如无法一次性完成需明确说明原因和后续计划
- 所有业务实现必须编写清晰的注释和日志记录
- 优先保证代码质量和可维护性，而非快速完成

##  安全规范

### 敏感信息处理
- 严禁在代码中硬编码任何密钥、密码、Token 等敏感信息
- 所有敏感配置必须从配置文件或环境变量中读取
- 日志中禁止打印完整的用户密码、信用卡号等敏感信息
- 敏感数据输出时需进行脱敏处理

### 数据安全
- SQL 查询必须使用参数化查询，严禁字符串拼接
- 所有用户输入必须经过严格校验，不得直接传入数据库查询
- 防止 SQL 注入、XSS 等常见安全漏洞

##  架构规范

### 接口设计
- 所有接口统一返回 `Result<T>` 包装类，禁止硬编码返回类型
- 保持 API 响应格式一致性，便于前端统一处理
- 合理的 HTTP 状态码使用

### 代码组织
- Controller 层仅负责接收请求和返回响应
- 参数校验逻辑放在 Service 实现类中，避免在 Controller 中写 if 判断
- 遵循单一职责原则，各层职责清晰

##  编码规范

### 命名规范
- 方法名以动词开头：`getUserById`、`createOrder`、`deleteById`
- 类名使用 PascalCase，方法名和变量名使用 camelCase
- 常量使用 UPPER_SNAKE_CASE
- 包名使用小写字母，多个单词用点分隔

### 注释规范
- 所有公开方法必须包含完整的 Javadoc 注释
- 注释应说明方法用途、参数含义、返回值和可能抛出的异常
- 避免无意义注释（如 `// 获取用户`、`// 返回结果`）
- 复杂业务逻辑需添加详细的实现说明

### 代码实践
- 批量操作使用 `saveAll` / `deleteAllById` 等方法，禁止循环单条操作
- 合理使用设计模式，提高代码复用性
- 异常处理要明确，避免空 catch 块
- 资源使用后及时释放，使用 try-with-resources 或 finally 块

##  日志规范

### 日志级别
- ERROR：系统错误、异常情况，需要立即关注
- WARN：潜在问题，可能影响系统运行
- INFO：重要业务流程节点、关键状态变更
- DEBUG：调试信息，详细执行过程

### 日志内容
- 业务操作需记录关键参数和执行结果
- 异常信息需包含完整堆栈跟踪
- 使用结构化日志，便于后续分析
- 日志输出要适度，避免过多影响性能

##  性能规范

### 数据库操作
- 合理使用索引，避免全表扫描
- 批量操作替代循环单条操作
- 分页查询处理大数据量
- 避免 N+1 查询问题

### 内存管理
- 及时释放不需要的对象引用
- 大对象处理注意内存占用
- 合理使用缓存机制

##  测试规范

### 单元测试
- 核心业务逻辑必须编写单元测试
- 测试用例覆盖正常流程和异常场景
- 使用 Mock 技术隔离外部依赖

### 集成测试
- 接口层面进行集成测试验证
- 数据库操作进行集成测试
- 确保测试环境的独立性

##  代码审查要点

### 必查项
- [ ] 是否遵循了上述所有规范
- [ ] 是否存在安全隐患
- [ ] 是否有适当的异常处理
- [ ] 日志记录是否完整合理
- [ ] 性能是否存在明显问题

### 建议项
- 代码是否可以进一步简化
- 是否有更好的实现方案
- 文档是否同步更新
